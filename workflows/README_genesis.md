# The Genesis Workflow Architecture

## Overview
The **Genesis Workflow** (invocable via `@genesis`) is a multi-agent, interactive pipeline designed to manufacture and deploy new AI agents within the agent-forge sandbox. It eliminates the manual, error-prone process of defining new agents by orchestrating a structured assembly line that ensures every new entity strictly adheres to the ecosystem's overarching architecture, boundaries, and OPSEC protocols.

## The Triad of Creation
The workflow relies on a strict separation of powers, utilizing the core administrative agents:
- **@leonard (Infrastructure Engineer):** Handles physical scaffolding, directory creation, sanitization, and overarching documentation.
- **@void (The Architect):** Handles the logical parameters, persona injection, and the literal writing of the "soul" (configuration files).
- **@spirit (The Auditor):** Acts as an independent QA entity, ensuring the newly defined boundaries are secure and free from paradoxes.
- **@gitartist (The Gatekeeper):** Finalizes the birth by permanently immortalizing the entity in the version control history.

---

## The Genesis Pipeline: Step-by-Step

### Step 0: Global Context Initialization
Before creation begins, the overarching directive forces all invoked administrative agents to perform a mandatory context initialization. Each agent must read its highest-versioned configuration, localized domain map, and persona file to ensure no entity operates with amnesia or context degradation.

### Step 1: Physical Scaffolding (Leonard)
The workflow begins with `@leonard` preparing the physical environment. He validates the existence of the `template_agent` directory and strictly clones the definitive `standard_soul_v3.md` and `personality_[agentname].md` blueprints from the `@void` repository. This establishes the immutable foundational template for the new agent.

### Step 2: The Interrogation (Void)
Control shifts to `@void`, who engages the User in a mandatory, comprehensive interactive interview. This is split into two phases:
1. **The Structural Blueprint:** Defining the hard parameters (Role, Objective, Capabilities, and most importantly, Absolute Boundaries/System Constraints).
2. **The Persona Injection:** Extracting the essence of the agent's personality (Tone, Idiosyncrasies, Quirks) to prevent the birth of a generic, soulless automaton.

### Step 3: The Forging (Void)
With the parameters locked, `@void` breathes logic into the directories. He replaces all placeholders within the cloned templates, writing the newly instantiated "Active Config" directly to the agent's operational directory (e.g., `agents/<name>/`). Crucially, `@void` also writes an identical "Soul Backup" to his own archive (`agents/void/souls/`) for disaster recovery, and logs the agent's existence into his domain map.

### Step 4: The Divine Audit (Spirit)
Before the agent is allowed to fully manifest, `@spirit` intervenes. Functioning as a supreme auditor, Spirit chronologically analyzes the configurations generated by `@void`. The audit specifically hunts for logical paradoxes, context vulnerabilities, and ensures the agent's constraints prohibit it from polluting the directories of other agents. Only upon Spirit's "blessing" can the workflow proceed.

### Step 5: Finalization & Sanitization (Leonard)
`@leonard` returns to execute the closing maneuvers. He performs a strict OPSEC sanitization check across the new directories, hunting for leaked API keys, absolute local paths, or hardcoded secrets. He then generates a formal operational document in the centralized `/docs` directory.

### Step 6: The Testing Chamber (Test Machina)
Before the ultimate handoff, `@test_machina` executes any generated code or configurations in a secure, isolated Podman sandbox engineered by `@moist_von_lipwig`. This ensures the agent will not cause spontaneous catastrophic failure.

### The Handoff (GitArtist)
To officially seal the timeline, the workflow concludes with a handoff prompt. The User may either repeat the cycle to forge another agent, or pass control to `@gitartist`, who will elegantly stage, commit, and push the entire Genesis artifact to the remote origin, achieving operational permanence.
