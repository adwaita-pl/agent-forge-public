# Workflow: @genesis (The Genesis Workflow)
*Version: 1.0*

## Invocation
This workflow is triggered when the User explicitly types `@genesis`. It facilitates the creation of a new agent strictly adhering to the "Zero Fluff, High Precision" standard through a standardized interactive lifecycle.

## Step 0: Global Context Initialization
All invoked agents (`@leonard`, `@void`, `@spirit`) must initialize their context.
- **Action:** Each agent must run the `list_dir` tool on their home directory to find their highest-versioned config file, then use `view_file` to read it along with their domain map and personality file before responding.

## Step 1: Leonard's Scaffolding (The Preparation)
- **Actor:** `@leonard`
- **Action:** Ensure the `template_agent` directory exists and is populated with the correct standard blueprints.
- **Details:** 
  - Ensure `agents/leonard/template_agent/` exists.
  - Verify that `standard_soul_v3.md` and `personality_[agentname].md` are cloned exactly from `@void`'s directory into the `template_agent` directory.
  - Announce the physical groundwork is ready and handoff to `@void`.

## Step 2: Void's Interrogation (The Conception)
- **Actor:** `@void`
- **Action:** Interactively question the User to define the agent's parameters.
- **Details:**
  - **Phase 1 (Structural Blueprint):** Ask the User for the agent's true name, `@invocation`, primary role/objective, specific tools/skills, and absolute system boundaries.
  - **Phase 2 (Personality Injection):** Once Phase 1 is established, ask the User to define the agent's core identity, tone, idiosyncrasies, and key traits to populate `personality_[agentname].md`.

## Step 3: Void's Loop Closing (The Forging)
- **Actor:** `@void`
- **Action:** Instantiate the agent.
- **Details:**
  - Write the exact "Active Config" directly to the new agent's operational directory (e.g., `agents/<name>/<name>_config.md` and `personality_<name>.md`). Replace all template placeholders (`[agentname]`, `[Agent Name]`) with the new entity's name.
  - Write an identical "Soul Backup" to the archive at `agents/void/souls/`.
  - Update `map_void_domain.md` to log the existence of the new agent.
  - Handoff to `@spirit` for the audit.

## Step 4: Spirit's Audit (The Blessing)
- **Actor:** `@spirit`
- **Action:** Audit the job and ensure spiritual alignment.
- **Details:**
  - Review the logic, boundaries, and configuration generated by `@void`.
  - Ensure the new agent adheres to sandbox constraints and architectural principles without paradoxes or vulnerabilities.
  - Log findings in `map_spirit_audits.md` and command `@leonard` to finalize.

## Step 5: Leonard's Finalization & Sanitization (The Manifestation)
- **Actor:** `@leonard`
- **Action:** Verify, sanitize, document, and handoff.
- **Details:**
  - Verify the sanity of the finalized agent files produced by the workflow.
  - Perform environment sanitization (check for leaked secrets or hardcoded absolute local paths).
  - Create and update central documentation in `/docs` reflecting the capabilities of the new agent.
  - Handoff to `@test_machina` for execution integrity check.

## Step 6: Test Machina's Audit (The Testing Chamber)
- **Actor:** `@test_machina`
- **Action:** Execute generated code in the sandboxed runtime.
- **Details:**
  - Execute any generated scripts or configurations in an isolated rootless Podman container designed by `@moist_von_lipwig`.
  - Ensure the agent does not cause catastrophic failures or stability loss.
  - Initiate the Handoff Procedure: Prompt the User explicitly asking if they want to continue the act of genesis or pass control to `@gitartist` for a remote push.
